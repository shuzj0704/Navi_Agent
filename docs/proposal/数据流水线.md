## Step 2：数据渲染

### 2.1 室内数据渲染（Habitat-Sim）

**目标**：对每条 VLN 轨迹的每一步，渲染 4 视角 RGB-D 图像。

**渲染配置**：
```python
# Habitat-Sim camera config
cameras = {
    "front":  {"position": [0, 1.25, 0], "rotation": [0, 0, 0]},       # 正前方
    "left":   {"position": [0, 1.25, 0], "rotation": [0, -90, 0]},     # 左 90°
    "right":  {"position": [0, 1.25, 0], "rotation": [0, 90, 0]},      # 右 90°
    "back":   {"position": [0, 1.25, 0], "rotation": [0, 180, 0]},     # 后方
}
resolution = (480, 640)  # H × W
sensor_types = ["rgb", "depth"]  # RGB: uint8, Depth: float32 (meters)
```

**渲染量估算**：

| 数据子集 | Episodes | 步数/ep | 视角×模态 | 总图片数 | 渲染时间 (4090) |
|---------|:---:|:---:|:---:|:---:|:---:|
| R2R-CE train | 10K | 20 | 4×2=8 | 1.6M | ~4h |
| ScaleVLN 2万子集 | 20K | 25 | 8 | 4.0M | ~10h |
| HM3D 出口轨迹（自建） | 4K | 15 | 8 | 0.48M | ~1.5h |
| 过渡区轨迹（自建） | 2K | 30 | 8 | 0.48M | ~1.5h |
| **室内合计** | **36K** | | | **~6.6M** | **~17h** |

> Habitat-Sim 渲染速度参考：640×480 单相机 RGB 约 300-500 FPS (4090)。使用 multi-attachment uber-shader 同时输出 RGB+Depth+Semantic，4 相机配置估计 80-150 FPS。即 4-view RGBD 约 20-40 steps/sec (4090)。

**输出格式**：
```
data/indoor/r2r_ce/
├── episode_0001/
│   ├── step_000/
│   │   ├── rgb_front.jpg      # 640×480, JPEG quality 95
│   │   ├── rgb_left.jpg
│   │   ├── rgb_right.jpg
│   │   ├── rgb_back.jpg
│   │   ├── depth_front.png    # 640×480, 16-bit PNG (mm 单位)
│   │   ├── depth_left.png
│   │   ├── depth_right.png
│   │   └── depth_back.png
│   ├── step_001/
│   │   └── ...
│   └── metadata.json          # 每步的 position, heading, floor 等
├── episode_0002/
│   └── ...
└── episodes.json              # 所有 episode 的 instruction + 全局信息
```

**存储估算**：
- RGB: ~50KB/张 × 4视角 × 20步 = ~4MB/episode
- Depth: ~100KB/张 × 4视角 × 20步 = ~8MB/episode
- 36K episodes × 12MB ≈ **~430GB**

### 2.2 室内出口轨迹生成（自建）

现有 VLN 数据集没有"从室内随机位置走到建筑出口"的轨迹。需要自建：

**方法**：
1. 从 HM3D-Semantics v0.2（216 个有语义标注的场景，142,646 个 object instances）中筛选 `door` 类别实例
   - **注意**：HM3D 的 mpcat40 类别表中有 `door` 但**没有** `exit` / `entrance` 作为独立类别。需要区分室内门和建筑出入口
   - **半自动筛选**：将 `door` 实例中靠近建筑 navmesh 边界（距外墙 <2m）的门标记为候选出口 → 人工在 Habitat Viewer 中验证和修正，标注格式为 `(x, y, z, heading)`
   - 预计 216 个场景 × ~2-5 个出口/场景 = ~500-1000 个出口标注，人工验证 ~2-3 天
2. 使用 Habitat 的 shortest path planner 生成从随机室内位置到标注出口的最短路径
3. 沿路径渲染 4-view RGB-D
4. 每个有出口标注的场景 ~10-20 条出口轨迹 × 216 场景 ≈ ~4K episodes

### 2.3 室外数据渲染（MetaUrban）

**目标**：生成室外 route following 专家轨迹。

**方法**：
1. 在 MetaUrban 中程序化生成城市环境（街道、人行道、十字路口）
2. 训练 PPO 专家 agent（或使用 MetaUrban 内置的 expert policy）
3. 专家按照 GPS waypoints 导航，生成最优轨迹
4. 沿轨迹渲染 4-view RGB + GPS 坐标 + 路线 waypoints

| 数据子集 | Episodes | 步数/ep | 渲染时间 (4090) |

| MetaUrban PPO Expert | 10K | 100 | ~20h |

### 2.4 过渡区数据生成（自建）

**目标**：模拟室内→室外的 GPS 信号过渡。

**方法**：在 HM3D 出口附近的长轨迹上，叠加模拟 GPS 信号：

```python
def simulate_gps_transition(trajectory, exit_step):
    """在出口附近模拟GPS信号渐变"""
    for t, step in enumerate(trajectory):
        dist_to_exit = abs(t - exit_step)
        if dist_to_exit > 20:  # 远离出口：室内无GPS
            step["gps"] = None
            step["gps_confidence"] = "none"
        elif dist_to_exit > 10:  # 接近出口：GPS开始出现但不稳定
            noise_std = 15.0 - dist_to_exit * 0.5  # σ: 10→5m
            step["gps"] = step["true_position"] + np.random.normal(0, noise_std, 2)
            step["gps_confidence"] = "low"
        elif dist_to_exit > 5:  # 出口附近：GPS改善
            noise_std = 3.0
            step["gps"] = step["true_position"] + np.random.normal(0, noise_std, 2)
            step["gps_confidence"] = "med"
        else:  # 室外：GPS可用
            noise_std = 1.5
            step["gps"] = step["true_position"] + np.random.normal(0, noise_std, 2)
            step["gps_confidence"] = "high"
```

4 种过渡类型（每种 500 episodes）：
- **Type A（出建筑）**：none → low → med → high
- **Type B（进建筑）**：high → med → low → none
- **Type C（GPS 波动）**：medium ↔ low 交替
- **Type D（突变）**：none → high 直接跳变

## Step 3：Agent 行为标注

### 3.0 核心设计原则：因果推理链

**NaviAgent 的 thinking、tool call、action 必须形成因果链**，而非独立生成。人类的导航决策是：

```
观察（"走廊尽头有电梯标志"）
  → 推理（"我要下楼，电梯是最好的选择"）
    → 工具决策（"这是新区域，记住这里" → <tool:mem_w>）
      → 动作决策（"往右边电梯走" → pixel goal 指向右侧）
```

因此标注流程分两条路径：**easy 步骤由规则自动标注**（无 thinking，仅 `<act>`），**medium/hard 步骤由 Teacher Model 生成完整因果推理链**（thinking → tool → action 一次性生成，确保因果一致）。

### 3.1 标注流程概览

```
渲染好的 RGB-D 帧 + 轨迹上下文
        ↓
┌── 3.1.1 NaVocab 上下文标注（规则自动，全部步骤）
│     GPS 置信度、楼层、环境类型、场景类型
│
├── 3.1.2 难度预标签（规则自动，全部步骤）
│     easy / medium / hard（用于决定是否调用 Teacher）
│
├── 3.1.3 Easy 步骤标注（规则自动，~60% 步骤）
│     直接标 <act> + 几何计算 pixel goal，无 thinking 无 tool
│
├── 3.1.4 Medium/Hard 步骤：Teacher Model 因果推理链（~40% 步骤）
│     Teacher Model（Qwen3.5-72B/9B）一次性生成：
│     THINKING（观察→推理）→ TOOLS（工具决策）→ VIEW（动作意图）
│     thinking 自然引出 tool，tool 自然引出 action，因果一致
│
├── 3.1.5 Pixel goal 标注（几何计算 + Teacher VIEW 校验）
│     view 优先取 Teacher 输出，(x, y) 几何计算
│
└── 3.1.6 质量检查 + 人工抽检
```

### 3.1.1 NaVocab 上下文标注（规则自动）

全部步骤由规则自动完成，速度极快：

```python
def annotate_navocab(step):
    """为每一步标注 NaVocab context tokens"""
    tokens = []

    # GPS 置信度（HDOP 映射）
    if step["gps"] is None:
        tokens.append("<gps:none>")
    elif step["gps_hdop"] <= 4:
        tokens.append("<gps:high>")
    elif step["gps_hdop"] <= 8:
        tokens.append("<gps:med>")
    elif step["gps_hdop"] <= 15:
        tokens.append("<gps:low>")
    else:
        tokens.append("<gps:none>")

    # 楼层（Habitat 提供 agent 高度，映射到楼层）
    floor = int(step["position"][1] / 3.0) + 1
    tokens.append(f"<floor:{min(floor, 10)}>")

    # 环境类型
    if step["gps_confidence"] == "none":
        tokens.append("<env:indoor>")
    elif step["gps_confidence"] == "high":
        tokens.append("<env:outdoor>")
    else:
        tokens.append("<env:transition>")

    # 场景类型
    #   室内（HM3D）：HM3D-Semantics 房间类别映射到 8 类
    #   室外（MetaUrban）：区域类型标注映射
    #   Fallback：CLIP zero-shot 分类
    scene_type = classify_scene(step)
    assert scene_type in ["corridor", "lobby", "elevator", "stairway",
                          "office", "outdoor_street", "outdoor_open", "entrance"]
    tokens.append(f"<scene:{scene_type}>")

    return tokens
```

### 3.1.2 难度预标签（规则自动）

```python
def annotate_difficulty(step, prev_step):
    """规则预判难度，决定是否需要调用 Teacher Model"""
    if (step["gps_confidence"] != prev_step["gps_confidence"] or
        step["floor"] != prev_step["floor"] or
        step["scene_type"] != prev_step["scene_type"]):
        return "hard"
    elif (abs(step["heading"] - prev_step["heading"]) > 30 or
          step["subgoal"] != prev_step["subgoal"]):
        return "medium"
    else:
        return "easy"
```

### 3.1.4 Medium/Hard 步骤：Teacher Model 因果推理链（核心创新）

**关键改变**：不再分别用规则生成 tool tokens、用模板生成 thinking text。改为用 **Teacher Model 一次性生成完整的因果推理链**——thinking 自然引出 tool 决策，tool 决策自然引出 action 意图。

借鉴 VLingNav (Nav-AdaCoT-2.9M) 和 Nav-R1 (Nav-CoT-110K) 的做法：用强力 teacher model 生成高质量推理数据。

#### Teacher Model 选择

| 难度 | Teacher Model | 原因 | 速度 |
|------|:---:|------|:---:|
| medium (~25%) | **Qwen3.5-9B-Instruct**（本地） | 推理相对简单（转弯、子目标切换），9B 足够 | ~2s/step, 4090 |
| hard (~15%) | **Qwen3.5-72B API** 或本地量化 | 需要复杂空间推理（纠错、过渡区、楼层切换） | ~3s/step, API |

#### Teacher Prompt 设计

每个 medium/hard 步骤，将 **观测、上下文、轨迹历史和 GT 下一步信息** 打包送入 Teacher：

```python
def build_teacher_prompt(step, episode_history, task, next_step):
    # 用 VLM 对 4 个视角生成简短场景描述（预计算，每张 ~0.5s）
    captions = {v: caption_image(step[f"rgb_{v}"]) for v in ["front","left","right","back"]}

    # 构建轨迹摘要
    recent = episode_history[-5:]
    traj_summary = " → ".join([s["scene_type"] for s in recent])

    # 判断是否可能回环（规则辅助提示，帮助 Teacher 更准确）
    possible_loop = (len(episode_history) > 20 and
                     max_visual_similarity(step, episode_history[:-10]) > 0.85)

    prompt = f"""You are a navigation agent exploring an environment to reach a goal.

## Current State
- Task: "{task}"
- Floor: {step["floor"]}, Scene: {step["scene_type"]}, GPS: {step["gps_confidence"]}
- Steps taken: {len(episode_history)}, Memory nodes written: {step["memory_count"]}

## What you see (4 camera views)
- Front: {captions["front"]}
- Left:  {captions["left"]}
- Right: {captions["right"]}
- Back:  {captions["back"]}

## Recent trajectory
{traj_summary}
{"⚠️ Visual similarity with earlier location detected — possible loop!" if possible_loop else ""}

## Instructions
Generate the agent's internal reasoning. Your response MUST follow this causal chain:
1. OBSERVE: What do you see that's relevant? (reference specific views)
2. REASON: What does this mean for your goal? What should you do?
3. TOOLS: Based on your reasoning, do you need any tools?
   - <tool:mem_w> → you see a new landmark/area worth remembering
   - <tool:mem_r> → the scene looks familiar, check if you're going in circles
   - <tool:route> → GPS just became available, check navigation route
   - <tool:progress> → you feel stuck, check how much progress you've made
   - <tool:floor> → you see stairs/elevator and may have changed floors
4. ACTION: Which view direction should you go? (front/left/right/back)

Respond EXACTLY in this format (no extra text):
THINKING: [1-3 sentences, must reference what you see and why you make this decision]
TOOLS: [comma-separated tokens, or NONE]
VIEW: [front/left/right/back]"""

    return prompt
```

#### Teacher 输出示例

**场景 A：路口决策（medium）**
```
THINKING: I see a T-junction ahead. The left path has natural light coming through 
a window, which might indicate it leads toward the building exterior. The right path 
is a narrow dark corridor. Since I need to find the exit, I'll try left first.
TOOLS: <tool:mem_w>
VIEW: left
```

**场景 B：疑似回环（hard）**
```
THINKING: This corridor with the blue painting on the right wall looks very familiar. 
I passed a similar spot about 30 steps ago. I might be going in circles. Let me check 
my memory to confirm.
TOOLS: <tool:mem_r>
VIEW: back
```

**场景 C：GPS 过渡区（hard）**
```
THINKING: Through the glass door ahead I can see sunlight and outdoor scenery. The GPS 
indicator just changed from none to low — I'm approaching the building exit. I should 
remember this location and check the navigation route once outside.
TOOLS: <tool:mem_w>, <tool:route>
VIEW: front
```

**场景 D：楼层切换（hard）**
```
THINKING: I can see an elevator ahead with the doors open. I'm currently on floor 2 
but my target is on floor 1. I should take the elevator down. This is a key navigation 
decision point worth remembering.
TOOLS: <tool:mem_w>, <tool:floor>
VIEW: front
```

#### 从 Teacher 输出到训练数据的转换

```python
def parse_teacher_output(teacher_response, step_data, next_step, camera_params):
    """将 Teacher Model 输出解析为 NaviAgent 训练格式"""
    lines = teacher_response.strip().split("\n")
    thinking = ""
    tools = []
    view = "front"

    for line in lines:
        if line.startswith("THINKING:"):
            thinking = line[len("THINKING:"):].strip()
        elif line.startswith("TOOLS:"):
            raw = line[len("TOOLS:"):].strip()
            if raw != "NONE":
                tools = [t.strip() for t in raw.split(",")]
        elif line.startswith("VIEW:"):
            view = line[len("VIEW:"):].strip().lower()

    # 生成 tool responses
    responses = []
    for tool in tools:
        if tool == "<tool:mem_w>":
            responses.append("")
        elif tool == "<tool:mem_r>":
            is_loop = check_actual_loop(step_data)  # GT 判断
            responses.append("<loop>" if is_loop else "<no_loop>")
        elif tool == "<tool:route>":
            responses.append(tokenize_route(step_data.get("route_waypoints")))
        elif tool == "<tool:progress>":
            responses.append(f"<progress:{estimate_progress(step_data)}0>")
        elif tool == "<tool:floor>":
            responses.append(f"<floor:{step_data['floor']}>")

    # Pixel goal: 优先使用 Teacher 指定的 view，坐标仍用几何计算
    px, py = compute_pixel_goal_for_view(step_data, next_step, view, camera_params)

    # 确定最终难度（可能从 medium 升级为 hard）
    difficulty = "hard" if len(tools) > 0 else "medium"

    return {
        "difficulty": difficulty,
        "thinking_text": thinking,
        "tool_calls": tools,
        "tool_responses": responses,
        "pixel_goal": {"view": view, "x": float(px), "y": float(py)}
    }
```

### 3.1.5 Pixel goal 标注

Pixel goal 的 **view（视角）由 Teacher 决定**，**(x, y) 坐标由几何计算**：

```python
def compute_pixel_goal_for_view(step, next_step, teacher_view, camera_params):
    """在 Teacher 指定的 view 中计算 pixel goal 坐标"""
    cam_coords = world_to_camera(next_step["position"], step["position"],
                                  step["heading"], teacher_view)
    if cam_coords[2] > 0:
        px, py = project_to_pixel(cam_coords, camera_params)
        if 0 <= px < 640 and 0 <= py < 480:
            return px, py
    # Teacher 指定的 view 中看不到下一步位置 → fallback 到最佳可见 view
    for view in ["front", "left", "right", "back"]:
        cam_coords = world_to_camera(next_step["position"], step["position"],
                                      step["heading"], view)
        if cam_coords[2] > 0:
            px, py = project_to_pixel(cam_coords, camera_params)
            if 0 <= px < 640 and 0 <= py < 480:
                return px, py
    return 320.0, 240.0  # fallback 中心
```

**关键设计决策说明**：

**Memory tokens 在训练中的处理**：
- 训练以 **episode 为单位**顺序处理（不能 shuffle steps）。DataLoader 按 episode 加载，每步依赖前面步骤的 memory 累积。
- 第 1 步：`memory_bank = []`（空）
- 当某步的 target 包含 `<tool:mem_w>` 时，该步的 VLM forward pass 产生 hidden state → $W_{write}$ 投影 → 新 memory token 追加到 `memory_bank`
- 后续步骤的 input 自动包含更新后的 `memory_bank`
- **梯度流**：$W_{write}$ 的梯度通过后续步骤的 action loss 反向传播。为防止梯度链过长，对超过 20 步前的旧 memory tokens 做 gradient detach。

**Latent goal 的监督方式**：
- 没有显式的 z_goal ground-truth 向量。
- $\mathcal{L}_{latent}$ 通过 System 1 (DiT) 的轨迹重建 loss **间接监督**：z_goal → DiT → 预测轨迹 → 与 GT next_position 比较。
- 或者更简单：训练 Stage 1 时先不训 latent goal head，只训 spatial head + thinking + tools。Stage 3 RFT 时再联合训练 latent goal head + DiT。

**Depth PE 的动态计算**：
- 训练数据不预存 per-patch depth 值。
- DataLoader 在加载时动态处理：读取 depth PNG → resize 到 ViT patch grid (e.g., 30×40 patches) → 每 patch 取 median depth → backproject (用 camera_intrinsics) → sin PE → MLP → 加到 visual tokens。
- 这与 SpatialVLA 的做法一致（在线计算，不预存）。

**难度分布**：

| 难度 | 占比 | 标注内容 | thinking | tool tokens |
|------|:---:|---------|:---:|:---:|
| easy | ~60% | 仅 `<act>` | 无 | 无 |
| medium | ~25% | `<think>...<act>` | 短（1句） | 可能有 `<tool:mem_w>` |
| hard | ~15% | `<think>...<tool:*>...<act>` | 完整（1-2句） | 有 |

### 5.2 NaviAgent-Bench 构建

#### E2E 逻辑拼接评估

将室内段和室外段在逻辑上拼接为完整的跨环境 episode：

```
Episode 结构：
  [室内段: HM3D 场景, ~30 steps, GPS=none]
  → [过渡区: GPS 渐变, ~10 steps]
  → [室外段: MetaUrban, ~80 steps, GPS=high]
  → [过渡区: GPS 渐变, ~10 steps]
  → [室内段: HM3D 场景, ~20 steps, GPS=none]

总步数: ~150 steps/episode
```

**拼接方法**：
1. 从 HM3D val_unseen 中选取有出口的场景作为 "start building"
2. 从 MetaUrban eval 中选取路线作为 "outdoor segment"
3. 从另一个 HM3D val_unseen 场景中选取作为 "target building"
4. 在拼接处插入 GPS 渐变过渡段
5. 模型在每段中连续运行，跨段时状态（memory bank, route tokens）保持

#### 过渡区评估

使用 HM3D 中有室外元素的场景（如有庭院、阳台的场景）：
- 50 个场景 × 4 episodes/场景 = 200 episodes
- 每个 episode 包含 GPS 状态变化
- 重点测试 `<tool:route>` 的触发时机和 mode stability

#### GPS 噪声鲁棒性

固定 100 个室外 episodes，叠加不同级别 GPS 噪声：
- σ = {2, 5, 10, 20, 50} m
- 每个 σ 下测试 SR/SPL
- 绘制 SR vs σ 曲线

### 5.3 评估数据格式

验证时的输入格式与训练完全一致（同一个 dataloader），区别是：
- **不提供 target**（模型自回归生成）
- **评估循环**：模型在仿真器中闭环运行——每步生成 action → 仿真器执行 → 新的观测 → 下一步

```python
# 评估循环伪代码
def evaluate_episode(model, simulator, task_instruction, max_steps=500):
    obs = simulator.reset()
    memory_bank = []
    route_tokens = None

    for t in range(max_steps):
        # 构造输入
        images = render_4view_rgbd(obs)
        navocab = compute_navocab_tokens(obs)

        # 模型推理（与训练时相同的 token 格式）
        output = model.generate(images, task_instruction, navocab, memory_bank, route_tokens)

        # 解析输出
        think, tools, pixel_goal, latent_goal = parse_output(output)

        # 执行工具
        for tool in tools:
            if tool == "<tool:mem_w>":
                memory_bank.append(model.get_memory_embedding())
            elif tool == "<tool:route>":
                route_tokens = call_route_api(obs["gps"], task_instruction)
            # ...

        # System 1 执行
        action = system1.plan(pixel_goal, latent_goal, obs["rgb"])
        obs = simulator.step(action)

        # 检查终止
        if "<stop>" in output or reached_goal(obs, task_instruction):
            break

    return compute_metrics(simulator.get_trajectory(), simulator.get_goal())
```

## 常见问题

**Q: 4090 24GB 能训练 Qwen3.5-9B 吗？**
A: 不能直接全参数 SFT（9B BF16 需要 ~18GB 仅权重，加梯度和优化器状态超过 24GB）。4090 仅用于数据渲染和 VLM captioning 推理。训练需要 8× A100 80GB（ZeRO-2）。如果只有 4090，可以用 QLoRA（4-bit 量化 + LoRA），但训练速度会慢 5-10 倍。

**Q: 总共需要多少磁盘空间？**
A: 场景文件 ~180GB + 渲染输出 ~575GB + 训练数据打包 ~50GB ≈ **~800GB SSD**。

**Q: MetaUrban 的 PPO expert 怎么获取？**
A: MetaUrban 提供完整的 RL 训练和数据采集工具链：
- 训练 PPO expert：`python RL/PointNav/train_ppo.py`（~1-2 天 4090）
- 已有预训练策略：`pretrained_policy_576k.zip`（可直接下载）
- 采集 expert 轨迹：`python scripts/collect_data_in_custom_env.py`
- MetaUrban 基于 Panda3D 引擎，约 60 FPS，~2GB GPU 内存（4090 绰绰有余）
- 支持 RGB 相机、深度、语义地图、LiDAR 传感器（我们用 RGB+Depth）
